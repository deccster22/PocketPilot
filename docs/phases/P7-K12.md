# P7-K12 - Trade Hub Inline Knowledge Help Wiring

## Why This Phase Happened Now

`P7-K11` defined bounded term eligibility, profile/surface treatment, and anti-clutter rules for Trade Hub help wiring.

`P7-K12` is the first runtime rollout of that plan, implemented only after the KI4/KI5 knowledge import and K8/K9/K10 glossary architecture were already stable.

## Dependencies

- `PX-KI4` for Trade Hub and glossary topic-path availability in the active docs tree.
- `PX-KI5` for progressive beginner/intermediate/advanced posture continuity.
- `P7-K8`, `P7-K9`, and `P7-K10` for service-owned inline glossary and tuning architecture.
- `P7-K11` for first-rollout scope and no-link-soup policy.

## Terms Wired In This Rollout

First-rollout terms only:

1. `Stop-loss price`
2. `Target price`
3. one active risk-basis label only (`Risk amount` or `Risk percent`)
4. `Guardrails`

## Surfaces Touched

- Trade Hub Guardrail Preferences support copy (`Guardrails` help affordance).
- Risk Tool input labels for `Stop-loss price` and `Target price`.
- Risk Tool input label for one active risk-basis term only (`Risk amount` or `Risk percent`).

## Service-Owned Eligibility And Routing

- Added `services/knowledge/createTradeHubHelpAffordances.ts`.
- The seam is deterministic and bounded to first-rollout terms only.
- Routing remains service-owned with explicit glossary + full-topic destinations:
  - `docs/knowledge/glossary/stop-loss-price.md` and `docs/knowledge/trade-hub/stop-loss-price.md`
  - `docs/knowledge/glossary/target-price.md` and `docs/knowledge/trade-hub/target-price.md`
  - `docs/knowledge/glossary/risk-amount.md` and `docs/knowledge/trade-hub/risk-amount.md`
  - `docs/knowledge/glossary/risk-percent.md` and `docs/knowledge/trade-hub/risk-percent.md`
  - `docs/knowledge/glossary/guardrails.md` and `docs/knowledge/trade-hub/guardrails.md`
- `services/trade/fetchTradeHubVM.ts` and `services/risk/fetchRiskToolVM.ts` now expose prepared affordance metadata.
- `app/` remains render-only and consumes prepared affordances without term matching/ranking/topic derivation.

## Link-Soup Guardrails Applied

- No broad text scanning or arbitrary auto-linking.
- No repeated linking of the same term across dense numeric rows by default.
- Active risk-basis linking is single-term only (`Risk amount` or `Risk percent`, never both at once).
- Submission/readiness/handoff boundary wording remains plain and unlinked.

## What Was Deliberately Not Wired

- Entry price, prepared planning levels, position size, reward/risk, and guardrail sub-terms.
- Linking every repeated term occurrence.
- Tooltip-swarm behavior or wiki-surface expansion.
- Any risk calculation, guardrail evaluation, readiness, submission, or execution-boundary semantic changes.

## Preview-Check Requirement

Preview check required: **yes**.

Reason: this phase adds new visible/tappable affordances on live Trade Hub/Risk Tool surfaces, so readability and non-dispatch boundary clarity must be visually verified before final pass.

Suggested human preview scenarios:

1. Trade Hub beginner profile, top + plan-preview area.
2. Risk Tool with stop-loss price and target price labels visible.
3. Guardrail Preferences section.
4. Active risk-basis label area (`Risk amount` or `Risk percent`).
5. Advanced profile quick pass (lighter/default help treatment).

## Follow-Up (Future Phases)

- Measured expansion to additional Trade Hub terms only after first-rollout evidence.
- Guardrail preference help refinement after usage review.
- Strategy Navigator and concept-help reuse only after evidence and bounded rollout design.

## Recommendation Handling

### Adopt now

- Keep using the bounded first-rollout seam and profile-aware service ownership as the default pattern.

### Add to backlog

- Wider term coverage and additional surfaces after preview and clutter evidence.

### Decline for now

- UI-side matching/ranking, broad auto-link sweeps, tooltip swarms, or any help behavior that weakens non-dispatch boundaries.
