# P8-I11 - Compare-Window Expansion

## Why This Phase Happened Now

P8-I4 added one first compare-period seam.
P8-I5 through P8-I10 widened the same reflection family with period summaries, archive, Year in Review, export preparation and dispatch, and journal groundwork.

That made the next safe step clear:

- deepen the same Insights reflection lane instead of opening a separate analytics surface
- move from one recent-only comparison toward a small approved compare-window set
- keep compare-window selection, shaping, and honest unavailability in `services/`

P8-I11 happens now because PocketPilot already has enough interpreted history and reflection groundwork to support calmer bounded comparisons, but it is still too early for freeform range builders, scorecards, AI commentary, or a reporting center.

## What P8-I11 Added

P8-I11 adds five narrow refinements:

1. One canonical `ComparisonWindowVM` contract family in `services/insights/types.ts`.
2. One service-owned compare-window builder in `services/insights/createComparisonWindowVM.ts`.
3. One canonical compare-window fetch seam in `services/insights/fetchComparisonWindowVM.ts`.
4. One minimal subordinate `InsightsCompareScreen` that lets the user switch across a small approved window set while still rendering prepared values only.
5. One light Insights handoff that keeps compare windows inside the existing reflection lane rather than creating a new top-level hub.

The compare-window read path is now:

`EventLedger -> EventLedgerQueries -> interpreted history helpers -> createComparisonWindowVM -> fetchComparisonWindowVM -> InsightsCompareScreen`

## Compare-Window Rules Locked In Here

P8-I11 establishes these rules:

- one canonical compare-window contract only
- one canonical compare-window fetch seam only
- compare-window selection stays explicit and bounded
- `services/` own compare-window selection, comparison shaping, and honest unavailable states
- `app/` renders prepared compare-window values only
- low-difference windows remain valid outcomes, not failed comparisons
- compare wording stays calm, descriptive, and non-moralizing
- raw ledger details, provider/runtime metadata, score language, predictive language, and advice language stay out of user-facing comparison output
- unsupported boundaries, such as before-versus-after strategy change without an honest event spine, must stay explicitly unavailable

## What Compare-Window Expansion Was Added

The first approved compare-window set is intentionally small:

- `LAST_90_DAYS_VS_PREVIOUS_90_DAYS`
- `LAST_QUARTER_VS_PREVIOUS_QUARTER`
- `LAST_YEAR_VS_PREVIOUS_YEAR`
- `BEFORE_STRATEGY_CHANGE_VS_AFTER_STRATEGY_CHANGE` as an explicit unsupported selection until the interpreted history spine can support that boundary honestly

Each available compare window returns:

- one short framing summary
- a few prepared comparison items
- optional limitations when one side is drawn from lighter interpreted history

## What P8-I11 Deliberately Did Not Add

P8-I11 does not add:

- arbitrary custom date-range builders
- push notifications
- background polling
- AI-generated comparison commentary
- export sprawl
- scorecards, streaks, or gamified reflection
- performance grading
- broad dashboard redesign
- predictive or advice language
- a giant analytics center

## What Future Reflection Work Can Build On

Later `P8` work can safely extend this seam with:

- additional approved compare windows when the product has a real reason for them
- strategy-change comparisons once the event/history spine can record that boundary honestly
- calmer archive or journal handoffs into compare windows if the broader reflection lane needs them later
- richer but still bounded comparison context that continues to reuse interpreted history rather than raw ledger browsing

Those future steps should deepen the same prepared-contract seam rather than turning reflection into spreadsheet cosplay, grading, or performance theatre.

## Recommendation Review

- Adopt now: One canonical compare-window expansion because the P8-I4 through P8-I10 groundwork now supports bounded multi-window comparison without widening the product into analytics theatre.
- Add to backlog: Additional approved compare windows, archive-to-compare handoffs, and strategy-change comparison once an honest boundary spine exists, because they may be useful later but are not ready for this phase.
- Decline for now: Freeform range builders, AI commentary, scorecards, gamified recap, predictive comparison language, and broad analytics centers because they would push the reflection lane away from PocketPilot's calm doctrine.
