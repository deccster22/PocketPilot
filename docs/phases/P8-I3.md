# P8-I3 - Insights Detail / Archive Navigation

## Why This Phase Happened Now
P8-I1 created the first thin Insights history shelf.
P8-I2 added an Insights-specific revisit boundary and calm continuity.

That made the next safe step clear:
- keep the existing top-level Insights surface as the first shelf
- let users go one layer deeper into interpreted history
- do it through the same service-owned seams rather than exposing raw ledger exhaust

P8-I3 happens now because the history and continuity seams are stable enough to support one deeper shelf, but the product still is not ready for broader reflection tooling.

## What P8-I3 Added
P8-I3 adds five narrow refinements:

1. One canonical `InsightsArchiveVM` contract in `services/insights/types.ts`.
2. One service-owned archive/detail builder in `services/insights/createInsightsArchiveVM.ts`.
3. One canonical archive/detail fetch seam in `services/insights/fetchInsightsArchiveVM.ts`.
4. One minimal subordinate `InsightsDetailScreen` that renders prepared archive sections and detail entries only.
5. One light top-level navigation handoff from `InsightsScreen` into that deeper interpreted shelf.

Top-level read path remains:

`EventLedger -> EventLedgerQueries -> Since Last Checked -> OrientationContext -> createInsightsHistoryVM -> createInsightsContinuity -> fetchInsightsHistoryVM -> InsightsScreen`

Deeper archive read path is now:

`EventLedger -> EventLedgerQueries -> Since Last Checked -> OrientationContext -> createInsightsHistoryVM -> createInsightsContinuity -> createInsightsArchiveVM -> fetchInsightsArchiveVM -> InsightsDetailScreen`

## Interpreted Archive Rules Locked In Here
P8-I3 establishes these rules:
- one canonical archive/detail contract only
- deeper means richer interpretation context, not raw event exhaust
- `detailNote` adds calm meaning, not technical payload
- repetitive rows still compress toward fewer stronger archive notes
- archive grouping, ordering, and selected-section fallback stay in `services/`
- `app/` renders prepared sections and items only
- archive can honestly be unavailable when interpreted history is too thin

## What P8-I3 Deliberately Did Not Add
P8-I3 does not add:
- compare-period summaries
- archive search complexity
- journaling
- exports
- behaviour scoring
- recap systems
- push notifications
- background polling
- AI-generated reflections
- provider/runtime diagnostics in user-facing Insights
- a raw ledger browser
- inbox or unread semantics

## What Future P8 Work Can Build On
Later `P8` work can safely extend this seam with:
- richer interpreted archive sections if the product needs them
- calmer compare-period framing when it receives a dedicated phase
- broader reflection summaries that still build on the same prepared history family
- optional archive refinements that preserve interpretation-first service ownership

Those later steps should build on the current archive seam rather than replacing it with a generic log viewer or reflection suite.

## Recommendation Review
- Adopt now: Add one canonical deeper archive/detail seam because Insights now has enough interpreted history structure to support one calm second shelf without broadening scope.
- Add to backlog: Gentle archive search or broader reflection summaries because they may help later, but they would expand P8-I3 beyond navigational refinement.
- Decline for now: Raw ledger browsing, unread semantics, and behaviour-scoring overlays because they conflict with PocketPilot's calm interpreted-history posture.
