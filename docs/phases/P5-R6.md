# P5-R6 - Risk Basis Toggle and Risk-Per-Trade Context

Status: done  
Last updated: 2026-04-10

## Why This Phase Happened Now

P5-R1 through P5-R5 already established the canonical non-dispatching risk lane, the selected-plan handoff for prepared plan references, and one conservative strategy-owned publishing seam.

P5-R6 happened now because the Trade Hub and ProtectionPlan stack was finally stable enough to expose one explicit user-facing risk basis and one prepared risk-per-trade context seam without widening into execution, enforcement, or portfolio-analytics sprawl.

## What Risk-Basis And Risk-Context Support Was Added

- added one canonical `RiskBasis` family in `services/trade/types.ts`:
  - `ACCOUNT_PERCENT`
  - `FIXED_CURRENCY`
  - `POSITION_PERCENT`
- added one canonical `RiskBasisAvailability` contract plus one canonical `selectRiskBasis` seam in `services/trade/selectRiskBasis.ts`
- added one canonical `RiskPerTradeContext` contract plus one canonical builder in `services/trade/createRiskPerTradeContext.ts`
- added one shared prepared risk lane assembler in `services/trade/createPreparedTradeRiskLane.ts`
- threaded selected risk basis and prepared risk context through:
  - `fetchTradeHubVM`
  - `fetchTradePlanPreviewVM`
  - `fetchConfirmationSessionVM`
  - `TradeHubSurfaceModel`
  - `TradePlanPreview`
- added one calm Trade Hub proof path:
  - explicit risk-basis toggle rendering only when the prepared service contract says it is available
  - subordinate prepared risk-per-trade context copy in the Trade Hub preview lane
- used current selected-account context plus prepared plan references only
- kept all risk framing in `services/`

## Support-First Rules Preserved

- no order dispatch
- no hidden execution logic
- no enforcement-by-default
- no silent basis switching inside the app layer
- no app-side risk math
- no profit-seeking or excitement language
- no broad portfolio-risk analytics

The new context answers: how is this risk being framed?  
It does not answer: should you take this trade?

## What Was Deliberately Not Added

- no live execution coupling
- no auto-blocking risk guardrails
- no hard enforcement modes
- no cross-surface rollout outside the Trade Hub lane
- no VaR, portfolio heatmaps, or broader portfolio-risk analytics
- no gamified risk/reward visuals
- no broker-specific risk payload expansion
- no recommendation engine or advice copy

## What Future Work Can Build On

- richer account-aware defaults if a future settings seam makes user intent explicit
- broader prepared risk context once more honest stop/target producers exist
- optional follow-through into knowledge or coaching surfaces without changing the Trade Hub boundary
- future enforcement or portfolio analytics only through a separate explicit phase

## Verification

- added direct coverage for explicit basis availability, unsupported-request rejection, and unavailable states
- added direct coverage for prepared risk-per-trade context building and basis-driven context changes
- updated Trade Hub, preview, and confirmation-session tests so selected basis now flows through prepared summaries
- updated Trade Hub view tests so the app only formats prepared risk and sizing contracts and does not perform risk math
- `npm run verify` passes

## Recommendation Review

### Adopt now

- one canonical risk-basis selector contract: adopted now because basis truth belongs in `services/trade/`, not in the screen
- one canonical prepared risk-per-trade context builder: adopted now because the Trade Hub needed legible framing without app-side math
- one calm Trade Hub toggle path: adopted now because user-selected framing should be explicit before any later risk-lane expansion

### Add to backlog

- account-level preferred default risk basis: add to backlog because explicit user preferences deserve their own settings seam
- richer prepared context once more plan producers publish honest stop/target anchors: add to backlog because the current seam should stay conservative
- cross-surface reuse of this same risk lane outside Trade Hub: add to backlog because this phase intentionally proves the path in one lane only

### Decline for now

- hard risk enforcement or auto-blocking: decline for now because PocketPilot remains support-first, not paternalistic
- broad portfolio-risk analytics: decline for now because that would widen the phase far beyond the Trade Hub seam
- profit-framed or gamified risk/reward storytelling: decline for now because it conflicts with PocketPilot's calm posture
