# P9-S4 - Strategy Navigator Scenario Contrast Deepening

## Why This Phase Happened Now

`P9-S1` established the first calm Strategy Preview contract.
`P9-S2` proved that the same service-owned lane could add optional knowledge follow-through without becoming onboarding or advice.
`P9-S3` then added one worldview-level explanation seam that answers why a strategy cares in the selected simulated backdrop.

That makes `P9-S4` the next honest step:

- not a live simulator
- not a prediction layer
- not a strategy battle surface

Instead, `P9-S4` deepens the finite scenario layer itself and adds one calm answer to:

- what changes for this strategy in this kind of scenario?

This phase happens now because PocketPilot already has:

- one canonical preview VM
- one finite scenario catalog
- one prepared preview-focus shape
- one prepared explanation seam
- one stable app boundary that already renders prepared preview content only

## What P9-S4 Added

P9-S4 adds six narrow pieces:

1. One modest scenario-catalog refinement with explicit interpreted scenario traits.
2. One canonical preview-contrast contract in `services/strategyNavigator/types.ts`.
3. One canonical preview-contrast builder in `services/strategyNavigator/createStrategyPreviewContrast.ts`.
4. One Strategy Navigator builder path that now shapes preview interpretation, preview explanation, preview contrast, and preview knowledge follow-through together.
5. One minimal preview-card addition that renders a small "What changes in this scenario" section only when the service-owned seam says it is honestly available.
6. Tests and docs that lock scenario contrast into a calm, descriptive, educational posture.

The current path is:

`selected strategy + selected scenario + interpreted scenario traits + prepared preview focus -> createStrategyPreviewContrast -> createStrategyNavigatorVM -> fetchStrategyNavigatorVM -> Strategy Preview card`

## Descriptive Contrast Rules Locked In Here

P9-S4 establishes these rules:

- one canonical preview-contrast contract only
- services own scenario-trait and preview-contrast shaping
- app renders prepared contrast content only
- scenario catalog stays finite and deterministic
- contrast answers worldview and emphasis questions, not outcome questions
- contrast remains subordinate to the preview itself
- unavailable is better than filler
- no execution, broker, readiness, profit, or forecast semantics
- no raw signal arrays, provider/runtime metadata, or simulator sprawl
- no urgency language

## What P9-S4 Deliberately Did Not Add

P9-S4 does not add:

- live market simulation
- execution handoff
- broker integration
- order dispatch
- AI commentary
- prediction scoring
- side-by-side freeform strategy battle views
- strategy recommendation ranking
- onboarding integration
- push notifications or background polling

## What Future P9 Work Can Build On

Later `P9` work can extend this seam with:

- richer but still finite scenario coverage that reuses the same interpreted-traits model
- deeper strategy-aware contrast depth when the current contract proves useful
- broader preview-owned explanation and knowledge bridges that remain subordinate to the simulated preview
- additional Strategy Navigator presentation depth that still keeps `services/` in charge of wording and contrast logic

Those later steps should deepen this same service-owned lane instead of replacing it with a simulator, advice engine, or execution funnel.

## Recommendation Review

- Adopt now: One canonical scenario-contrast seam, because Strategy Preview now has enough prepared context to explain how emphasis changes without drifting into prediction, advice, or live-simulator behaviour.
- Add to backlog: Slightly richer scenario variety or broader contrast coverage once this finite catalog proves useful, because the first version should stay short, calm, and easy to audit.
- Decline for now: Freeform simulator tooling, strategy-versus-strategy battle views, and recommendation-style next-step guidance, because they would widen Strategy Preview beyond its descriptive product promise.
