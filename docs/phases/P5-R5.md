# P5-R5 - Strategy-Owned Prepared Stop/Target Publishing

Status: done  
Last updated: 2026-04-05

## Why This Phase Happened Now

P5-R1 through P5-R4 already established the canonical risk-tool seam, quote-assisted entry help, the selected-session handoff for prepared plan references, and one canonical producer helper inside `services/trade/`.

P5-R5 happened now because that helper path is stable enough to accept richer service-owned strategy/event context without widening the risk contract, changing selector precedence, or turning the risk lane into order-entry support.

## What Strategy-Owned Publishing Enrichment Was Added

- added `services/trade/createStrategyPreparedRiskReferences.ts` as the one canonical strategy/event interpreter for derived prepared risk references
- kept `PreparedTradePlanRiskReferences`, `ProtectionPlan`, `ConfirmationSession`, and the risk-tool selector contract unchanged
- kept explicit `MarketEvent.metadata.preparedRiskReferences` as the strongest producer input when grouped plan context already carries clear numeric references
- added one thinner strategy-owned publishing path through `MarketEvent.metadata.strategyPreparedRiskContext`
- `createPreparedRiskReferences` now resolves fields in this order:
  - explicit prepared plan field from grouped event metadata
  - strategy-owned derived field when one scoped, confirmed event context can support it honestly
  - existing confirmed primary-event entry fallback for actionable plans only
- current strategy-owned publishing is intentionally conservative:
  - `momentum_basics` can publish a prepared `stopPrice` from a scoped baseline reference when the event is confirmed and the grouped stop context agrees
  - `dip_buying` can publish a prepared `targetPrice` from a scoped baseline reference when the event is confirmed and the grouped target context agrees
- multi-symbol, estimated, conflicting, or otherwise weak strategy-owned context stays `null`
- `fetchConfirmationSessionVM`, `fetchRiskToolVM`, and `selectRiskReferences` keep the same boring precedence:
  - `USER_INPUT`
  - `PREPARED_PLAN`
  - `PREPARED_QUOTE`
  - `UNAVAILABLE`

## What Was Deliberately Not Added

- no broker APIs
- no order dispatch
- no execution payload construction in `app/`
- no one-tap or prefilled execution path
- no quote-derived stop or target suggestions
- no hidden automation or defaults
- no account-scoped preferences or settings
- no multi-target reward framing
- no leverage, margin, derivatives, or P/L simulation tooling
- no raw signal, provider, or runtime leakage into user-facing risk outputs

## What Future P5 Risk-Tool Work Can Build On

- additional strategy-owned prepared references when later service-owned event context can explain them explicitly
- richer entry publishing only when a later strategy/event seam can support it more honestly than the current fallback
- account-scoped prepared-reference preferences once a dedicated settings seam exists
- broader planning support such as multi-target framing only if a later phase widens scope on purpose

## Verification

- added direct helper coverage for strategy-owned stop/target publishing, ambiguity, weak-context null behavior, and explicit-reference precedence
- updated `createProtectionPlans` coverage for baseline-grounded stop/target integration and ambiguous strategy-owned stop behavior
- updated `fetchConfirmationSessionVM` coverage for selected-session handoff of strategy-owned prepared references plus no-leak behavior
- updated `fetchRiskToolVM` coverage for `PREPARED_PLAN` labeling and execution-neutral output with richer prepared stop support
- updated strategy and market-event coverage so the strategy-owned publishing context is explicit and deterministic before trade services consume it
- `npm run verify` remains the merge gate for this phase

## Recommendation Review

### Adopt now

- Strategy-owned baseline publishing markers in event metadata: adopted now because the service helper needs concrete interpreted support instead of implicit per-strategy guesswork.
- One canonical strategy/event publishing helper: adopted now because it keeps prepared stop/target truth in `services/trade/` instead of scattering it across plan producers.
- Conservative under-publishing for weak or mixed context: adopted now because trust matters more than filling every field.

### Add to backlog

- Additional strategy-owned prepared reference bases beyond baseline: add to backlog because future strategies may justify them, but this phase should stay thin.
- Account-scoped prepared-reference preferences: add to backlog because explicit user override already exists and settings are out of scope here.
- Multi-target or laddered reward framing: add to backlog because it may help later but would widen this focused support seam.

### Decline for now

- Quote-derived stop or target suggestions: decline for now because they would invent exits from thinner context than this phase allows.
- Direct execution prefills from prepared plan references: decline for now because the risk lane must remain support-only and non-dispatching.
- Broker-style advanced calculators: decline for now because they would blur calm planning support into order-entry sprawl.
