# P6-A3 - Alert Threshold / Profile Sensitivity Tuning

## Why This Phase Happened Now
P6-A1 established one canonical message-policy seam and made the message families explicit.
P6-A2 proved that the same seam could serve Snapshot, Dashboard, and Trade Hub without turning PocketPilot into a notification platform.

The next natural step was not broader rollout.
It was making the existing seam more explicit about when interpreted change should stay quiet, become a narrower alert, or disappear entirely across different user profiles.

This phase happened now to deepen consistency before any future alert-family expansion.

## What This Phase Added
- one small additive refinement to `services/messages/types.ts` for `MessageSensitivityProfile` and `AlertThresholdDecision`
- one canonical helper in `services/messages/applyMessageProfileTuning.ts`
- one tighter Snapshot alert path inside `services/messages/createMessagePolicyVM.ts`
- one unchanged canonical `fetchMessagePolicyVM` seam for Snapshot, Dashboard, and Trade Hub
- focused tests proving profile-aware tuning remains deterministic, calm, and service-owned

## Tuning Foundation Added
P6-A3 adds explicit tuning rules above the existing interpreted message candidates.

Current profile posture:
- `BEGINNER` maps to `GUIDED`
- `MIDDLE` maps to `BALANCED`
- `ADVANCED` maps to `DIRECT`

Current alert-threshold decisions:
- `KEEP_AS_ALERT`
- `DOWNGRADE_TO_BRIEFING`
- `SUPPRESS`

Current behavior:
- beginner keeps strong event-only change as a calm `BRIEFING` rather than a louder `ALERT`
- middle may receive `ALERT` for stronger interpreted change, but middling change prefers `BRIEFING`
- advanced gets tighter, more compact `ALERT` copy only when stronger thresholds are met
- advanced does not bypass semantic family distinctions or flood controls
- thin interpreted context still resolves to no message rather than filler copy

## Distinctions Preserved
P6-A3 keeps these lines explicit:
- `BRIEFING` remains quieter than `ALERT`
- `REORIENTATION` remains separate from `ALERT`
- `REFERRAL` remains a routing or fit note, not a stop
- `GUARDED_STOP` remains a product-boundary stop, not punishment
- thin or missing context still resolves to `UNAVAILABLE`, not notification theatre

Profile tuning is secondary.
Semantic family meaning remains primary.

## Deliberately Not Added
P6-A3 does not add:
- push notifications
- background polling
- inbox or unread systems
- badge counts
- notification-center plumbing
- user-editable threshold settings
- generic toast or banner infrastructure
- recommendation engines
- urgency escalation mechanics
- broader per-surface rollout beyond Snapshot, Dashboard, and Trade Hub
- execution or order dispatch

## What Future P6 Work Can Build On
Later `P6` work can build on this phase by:
- using richer interpreted inputs for threshold tuning while keeping the same seam
- refining per-surface eligibility only where a natural inline home already exists
- deepening calm message copy without moving classification into `app/`
- exploring additional policy-aware surfaces without inventing an inbox or notification center

Any future work should keep the same canonical message-policy seam and avoid drifting into a notification platform.

## Recommendation Review
### Adopt Now
- Explicit profile-sensitivity mapping: it keeps tuning readable and prevents ad hoc per-screen heuristics.
- Narrow downgrade-or-suppress alert decisions: they make "fewer, stronger messages" concrete without changing the family model.
- Compact advanced alert copy: it honors profile economy without expanding alert volume.

### Add To Backlog
- Richer interpreted alert inputs beyond the current event confidence and change magnitude: useful later, but not required for this thin tuning rung.
- Additional surface eligibility rules built on the same seam: valuable later, but P6-A3 deliberately stops at Snapshot, Dashboard, and Trade Hub.
- User-visible explanation of why a message was downgraded or suppressed: potentially helpful later, but currently unnecessary and could overcomplicate calm inline copy.

### Decline For Now
- Push, inbox, and badge mechanics: they conflict with PocketPilot's calm inline posture and expand scope into a notification platform.
- User-editable sensitivity controls: they would widen this phase into settings and preference UX before the service-owned policy is mature enough.
- Generic toast or banner systems: they would blur message-family meaning and encourage louder UI behavior.
