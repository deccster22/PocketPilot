# P5-R11 - Non-Blocking Guardrail Evaluation Status

## Why This Phase Happened Now

P5-R6 made the risk basis explicit and carried a prepared risk-per-trade context through Trade Hub. P5-R7 added prepared sizing and max-loss output. P5-R8 remembered an account-level preferred basis. P5-R9 added calm guidance for thin or unsupported inputs. P5-R10 remembered optional guardrail preferences per account.

P5-R11 happened now because the lane was finally stable enough to compare the selected plan against the enabled guardrail preferences without turning PocketPilot into a default-blocking enforcement system or a compliance maze.

## What Was Added

- one canonical `GuardrailEvaluation` contract family in `services/trade/types.ts`
- one canonical service-owned guardrail-evaluation builder in `services/trade/createGuardrailEvaluation.ts`
- one Trade Hub VM seam that resolves guardrail evaluation from the selected plan, prepared risk lane, account value, and saved preferences
- one calm Trade Hub proof path that shows prepared evaluation status without adding blocking behavior by default

## What Was Deliberately Not Added

- no live order dispatch
- no hard enforcement by default
- no auto-blocking guardrails
- no punitive cooldown enforcement
- no background polling
- no push notifications
- no portfolio-wide analytics or scorecards
- no broad settings center
- no app-owned evaluation logic
- no fear-language, shame-language, or compliance theatre

## What Future Guardrail Work Can Build On

- richer evaluation detail if the product ever needs it explicitly
- future enforcement only through a separate explicit phase
- additional optional guardrails if they remain calm, explicit, and account-scoped
- possible cross-surface reuse of the same prepared evaluation contract

## Verification

- added direct coverage for explicit default-off suppression, account-scoped isolation, unsupported surface states, near/outside/within evaluation states, partial context, and cooldown descriptiveness
- updated Trade Hub surface and screen-view tests so the new guardrail-evaluation contract stays prepared in services and rendered in the app without app-owned evaluation logic
- `npm run verify` remains the merge gate for this phase

## Recommendation Review

### Adopt now

- one canonical guardrail-evaluation contract: adopt now because the lane needs one explicit prepared status seam
- one service-owned evaluation builder / fetch seam: adopt now because the app should never own evaluation comparisons
- one calm Trade Hub proof path: adopt now because the lane should show chosen-guardrail status without introducing blocking behavior

### Add to backlog

- richer evaluation drill-downs: add to backlog because the first pass is intentionally compact
- durable persistence refinements behind the current preference seam: add to backlog because this phase only needs the deterministic comparison layer today
- cross-surface reuse of guardrail evaluation: add to backlog because this phase proves the path in Trade Hub first

### Decline for now

- enforcement-by-default: decline for now because PocketPilot stays support-first
- a broad settings maze: decline for now because it would fight the product's calm posture
- automatic blocking when guardrails are enabled: decline for now because the phase is explicitly non-enforcing
