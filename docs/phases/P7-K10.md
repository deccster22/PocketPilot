# P7-K10 - Inline Glossary Exposure Signals / Tuning Hooks

## Why This Phase Happened Now

`P7-K8` established the first narrow runtime inline glossary-help seams and `P7-K9` improved alias/matching quality while keeping rollout scope intentionally tight.

The next honest rung was to add a thin internal signal layer so future tuning can use real service-owned surfaced/acknowledged counts instead of guesswork, without adding user-facing analytics or expanding runtime glossary surfaces.

## What Was Added

- One canonical service-owned aggregate signal store in `services/knowledge/inlineGlossarySignalStore.ts`.
- One canonical service-owned recording seam in `services/knowledge/recordInlineGlossarySignals.ts` that normalizes counts by:
  - canonical `topicId`
  - canonical surface (`DASHBOARD` or `TRADE_HUB`)
  - canonical profile tier (`BEGINNER`, `INTERMEDIATE`, `ADVANCED`)
- One canonical service-owned summary seam in `services/knowledge/fetchInlineGlossarySignalSummary.ts`.
- Minimal threading through existing K8/K9 seams:
  - `createInlineGlossaryHelp` records surfaced signals when prepared blocks are issued.
  - `acknowledgeInlineGlossaryTerms` records acknowledged signals through the existing tap/ack path.

## What Was Deliberately Not Added

- No new user-facing analytics/debug surface.
- No external telemetry or network analytics calls.
- No new glossary rollout surfaces.
- No app-owned signal derivation.
- No execution gating, push, inbox, badge, or feed mechanics.
- No glossary content rewrite or external content import.

## Future Work This Enables

- Internal tuning and operator-facing analysis phases can read one canonical summary seam instead of inventing ad hoc counters.
- Future admin/debug views can consume the prepared aggregate summary when explicitly approved as a separate phase.
- Broader glossary rollout decisions can be informed by measured surfaced/acknowledged patterns while preserving the current calm, optional product posture.

## Recommendation Handling

- Adopt now: keep the internal aggregate signal seams and summary seam as the canonical K10 baseline.
- Add to backlog: future internal/admin analytics consumers and broader tuning work as separate scoped phases.
- Decline for now: user-facing analytics UI, network telemetry shipping, and glossary behavior expansion unrelated to tuning hooks.
