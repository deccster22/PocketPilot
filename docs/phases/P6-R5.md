# P6-R5 - Snapshot Orientation Cohesion

## Purpose
P6-R5 unifies Snapshot's subordinate orientation layer into one coherent service-owned scan story.

Snapshot keeps its sacred core triad:
- Current State
- Last 24h Change
- Strategy Status

Everything beneath that core remains subordinate.

## What This Phase Adds
- one canonical Snapshot briefing zone only
- one service-owned precedence rule between reorientation and Since Last Checked
- one prepared `SnapshotBriefingState` contract on the existing `SnapshotSurfaceVM` path
- one shared foreground-only rendering path for initial mount and app return

## Canonical Rule
- if reorientation is available and visible, it owns the Snapshot briefing zone
- if reorientation is available but hidden because the current cycle was dismissed, Snapshot does not fall through to a second card
- if reorientation is not available, Since Last Checked may use the briefing zone
- if neither is meaningful, the zone stays hidden

## Delivery Notes
- `services/orientation/createSnapshotBriefingState.ts` owns briefing precedence
- `services/snapshot/fetchSnapshotSurfaceVM.ts` exposes one prepared `briefing` contract
- `app/screens/snapshotScreenView.ts` reads that prepared contract only
- `app/screens/SnapshotScreen.tsx` renders one subordinate briefing block only

## Dismissal
- existing reorientation dismissal behavior remains unchanged
- Since Last Checked remains non-dismissible in this phase
- no generalized briefing-dismiss state machine was added
- no unread, badge, feed, or notification mechanics were added

## Non-Goals Preserved
- no Snapshot core redesign
- no raw signal leakage
- no background polling
- no push notifications
- no second Snapshot fetch path
- no app-owned briefing selection logic
