# P5-R3 - Prepared Plan Stop/Target References

Status: done  
Last updated: 2026-04-05

## Why This Phase Happened Now

P5-R1 established the canonical non-dispatching risk seam.
P5-R2 added source-tagged prepared references plus quote-assisted entry help, but it intentionally left richer prepared plan exits parked until the selected Trade Hub context had an honest service-owned place to carry them.

P5-R3 happened now because the confirmation-session seam is mature enough to carry optional prepared plan references without turning the risk tool into order entry.

## What Prepared-Plan Risk Support Was Added

- `ProtectionPlan` and `ConfirmationSession` now carry `preparedRiskReferences` as an explicit service-owned object or `null`.
- `fetchConfirmationSessionVM` sanitizes that object into an honest selected-plan context before the app sees it.
- `fetchRiskToolVM` now consumes that prepared plan context alongside prepared quote context and explicit user input through the existing canonical selector.
- explicit user values still win
- prepared plan references outrank prepared quote references when both are honestly available
- quote assistance still does not invent stop or target references

## What Was Deliberately Not Added

- no broker APIs
- no order dispatch
- no execution payload construction in `app/`
- no direct execution prefills
- no quote-derived exit suggestions
- no hidden defaults or automation
- no multi-target or laddered reward tooling
- no leverage, margin, derivatives, or P/L simulation surfaces

## What Future P5 Risk-Tool Work Can Build On

- richer plan producers that can honestly populate `preparedRiskReferences`
- additional calm planning support once a later phase explicitly widens the risk lane
- account-scoped risk preferences once a dedicated settings seam exists

## Verification

- added selector coverage for explicit stop/target override and prepared plan entry precedence over quote help
- added fetch coverage for prepared plan entry, stop, and target references flowing from selected trade context
- kept no-leak checks around provider/runtime and execution-path internals in risk outputs
- kept app rendering on prepared VM data only, with unavailable references staying visually quiet
- `npm run verify` remains the merge gate for this phase

## Recommendation Review

### Adopt now

- Explicit `preparedRiskReferences` on `ConfirmationSession`: adopted now because the risk tool needs one service-owned plan handoff instead of UI heuristics.
- Prepared plan precedence over prepared quote for the same field: adopted now because user trust improves when a real plan reference beats generic quote help.
- Quiet unavailable-reference rendering: adopted now because it keeps the surface calm when no honest prepared value exists.

### Add to backlog

- Plan producers that populate prepared risk references from richer strategy context: backlog because P5-R3 adds the handoff path, not new plan heuristics.
- Account-scoped prepared-reference preferences: backlog because explicit manual override already exists and settings are out of scope here.
- Multi-target reward framing: backlog because it may be useful later but would widen this thin support seam.

### Decline for now

- Quote-derived stop or target suggestions: decline for now because the phase must not invent exits from thin quote context.
- Direct execution prefills from prepared plan references: decline for now because risk support must remain non-dispatching and support-first.
- Broker-style advanced calculators: decline for now because they would blur calm planning support into order-entry sprawl.
