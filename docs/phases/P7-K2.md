# P7-K2 - Knowledge Topic Detail Surface

## Why This Phase Happened Now

`P7-K1` created the first knowledge shelf seam, but it deliberately stopped at the top-level library surface.

The normalized knowledge import and documentation pass now provide one canonical knowledge tree, so the next useful step is not broad contextual rollout yet.
It is one calm deeper topic view keyed by `topicId`.

That sequence matters:

- the shelf had to exist first
- the canonical docs tree had to exist first
- the app still needed to avoid becoming a markdown browser

`P7-K2` happens now because PocketPilot can finally deepen one topic at a time without inventing a second knowledge system or spraying contextual links across the whole app.

## What P7-K2 Added

`P7-K2` adds five connected pieces:

1. One canonical in-app knowledge catalog generated from the normalized docs register and topic files.
2. One canonical topic-detail contract in `services/knowledge/types.ts`.
3. One service-owned topic-detail builder and fetch seam.
4. One minimal subordinate topic detail screen under the existing Knowledge Library surface.
5. One library-to-topic navigation path that remains optional and non-gatekeeping.

The current path is:

```text
docs/knowledge + CONTENT_REGISTER
-> generate-knowledge-catalog
-> knowledgeCatalog
-> createKnowledgeLibraryVM
-> fetchKnowledgeLibraryVM
-> KnowledgeLibraryScreen
-> createKnowledgeTopicDetailVM
-> fetchKnowledgeTopicDetailVM
-> KnowledgeTopicScreen
```

## Optional And Non-Gating Rules

`P7-K2` preserves the product posture established in `P7-K1`.

Rules locked in this phase:

- the library shelf remains the entry point
- topic detail remains optional
- unavailable topic detail degrades honestly
- no other major surface depends on knowledge consumption
- app renders prepared sections and related-topic cards only
- no raw file paths or markdown source structure reach the UI

## What P7-K2 Deliberately Did Not Add

`P7-K2` does not add:

- contextual rollout across Dashboard, Snapshot, Trade Hub, or Insights
- a recommendation engine
- raw repo-file browsing
- direct markdown rendering in app
- quizzes, streaks, badges, or progression mechanics
- AI-generated knowledge summaries
- background polling or push notifications
- trading execution

## Why The Phase Is Narrow

The topic-detail surface needed to stay smaller than the full future knowledge family.

That is why `P7-K2` prefers:

- one canonical generated runtime catalog over drifting duplicate sources
- one subordinate topic route over multi-surface rollout
- prepared sections over markdown rendering
- related topics over recommendation machinery
- explicit unavailable states over filler

## What Future P7 Work Can Build On

Later `P7` work can build on this seam by:

- widening contextual rollout from strategies, signals, and events through service-owned eligibility rules
- adding richer media treatment where the canonical catalog honestly supports it
- adding smarter topic ordering or recommendations after the shelf and detail contracts are already stable
- broadening the visible shelf beyond the current `NOW` baseline without changing the app-owned rules

Those later phases should reuse the same catalog and fetch seams instead of inventing parallel knowledge paths.

## Recommendation Review

- Adopt now: service-owned topic detail over the canonical generated catalog, because it deepens the library without reintroducing raw-doc drift.
- Add to backlog: contextual knowledge links from Dashboard, Snapshot, and Trade Hub, because the detail route now exists but broad rollout still needs careful relevance rules.
- Decline for now: AI-generated topic summaries, because this phase is meant to preserve canonical authored meaning rather than invent new explanation text.
