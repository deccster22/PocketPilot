# P5-R15 - Prepared Reference Availability Rendering

## Why This Phase Happened Now

`P5-R13` introduced one canonical prepared stop/target availability seam.
`P5-R14` then normalized source labels, limitation copy, and unavailable wording on that same seam.

`P5-R15` happens now because those service-owned seams are stable enough to show a compact Trade Hub proof path in the selected-plan preview without changing ownership, widening into execution, or reframing the surface as a planner terminal.

## What Was Surfaced

- one compact prepared-planning-level block inside the existing Trade Hub Plan Preview lane
- available prepared stop/target rows rendered from service-owned labels, values, and source labels
- one compact limitation note rendered from service-owned limitation copy when present
- one quiet unavailable line rendered only when service-owned visibility policy says it is useful (thin context)
- no app-side stop/target math, source-confidence interpretation, or unavailable-copy authoring

## User-Facing Terminology Alignment

- user-facing copy now prefers `Entry price`, `Stop-loss price`, `Target price`, `Asset symbol`, and `Prepared planning levels`
- prepared-plan rows now use `Prepared stop-loss level` and `Prepared target level`
- thin-context unavailable copy is plain-language and calm:
  - `This setup does not provide enough context for prepared stop-loss or target levels yet.`
- Risk Tool helper copy now says:
  - `Use price levels to estimate position size and risk. These are planning inputs only. They do not place an order.`
  - `Your own values override any prepared planning levels.`
- prepared planning-support line now says:
  - `Optional planning context from the selected plan. Your own values remain authoritative.`
- Guardrail Preferences includes one calm comprehension line:
  - `Guardrails are optional account-level checks for planning. They can help flag risk limits or cooldown settings, but they do not place trades.`
- internal architecture names such as `PreparedTradeReferencesAvailability`, selector helpers, and service seams intentionally keep existing `reference` naming to avoid unnecessary contract churn

## What Was Deliberately Not Added

- no live execution or dispatch behavior
- no broker adapter changes or payload widening
- no multi-target or laddered reward framing
- no account preference surface for prepared references
- no settings UX
- no quote-assisted predictive stop/target expansion
- no recommendation/profit/opportunity framing
- no warning, badge, or blocking treatment for unavailable prepared references

## Preserved P5-R13/P5-R14 Behavior

- manual override precedence is unchanged
- prepared selector precedence is unchanged
- thin or ambiguous context remains unavailable instead of invented
- prepared source labels, limitation notes, and unavailable wording stay service-owned
- Trade Hub remains support-first and non-dispatching
- app remains render-only and deterministic once inputs are supplied

## What Future P5 Work Can Build On

- optional expansion of prepared-reference annotations on the same service-owned seam
- additional deterministic strategy-owned prepared reference sources with the same copy and availability posture
- other calm surface placements that reuse the same compact rendering posture and quiet unavailable behavior

## Verification

- updated `app/screens/tradePlanPreviewView.ts` mapping tests for available and thin-context unavailable states
- added placement tests to keep prepared references subordinate and compact in Trade Hub
- updated service tests for conservative unavailable-visibility policy ownership
- `npm run verify` remains the merge gate for this phase

## Recommendation Review

### Adopt now

- compact prepared-reference rendering inside existing Plan Preview hierarchy
- service-owned unavailable-visibility policy for calm non-blocking rendering
- app render-only pass-through posture for prepared reference copy and availability

### Add to backlog

- broader reuse of the same compact prepared-reference rendering pattern on other calm planning surfaces
- richer but still non-predictive prepared-reference annotations where service context can justify them

### Decline for now

- execution-path widening or order submission behavior
- recommendation/profit/prediction framing
- warning-wall, alert, badge, or blocker treatment for unavailable prepared references
