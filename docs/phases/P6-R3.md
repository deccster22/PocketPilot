# P6-R3 - Reorientation Dismiss Persistence

## Scope
P6-R3 adds minimal durable persistence for reorientation dismissal on Snapshot.

This phase keeps the existing service-owned visibility seam and does not expand reorientation into a notification framework.

## Delivered In This Phase
- a minimal persisted dismissal shape with `dismissedAt`
- a narrow dismissal store seam with `load`, `save`, and `clear`
- Snapshot reloads the persisted dismissal state on app restart and feeds it into the existing prepared VM path
- services continue to own whether the prepared summary is visible or hidden
- stale dismissals are cleared when a later eligible reorientation summary is generated

## Behaviour Rules
- foreground-only
- one canonical placement surface only
- no broker APIs
- no order dispatch
- no push notifications
- no background polling
- no inbox/feed/badge system
- no reminder cycle
- no raw signal leakage

## Dismissal Rule
Dismissal remains surface behavior only.

Persisted dismissal hides the current prepared reorientation cycle across app restarts.
It does not delete the underlying summary contract.

P6-R3 resets dismissal when a newer eligible summary is generated from a later `lastActiveAt` boundary.
That keeps dismissal calm and finite instead of turning it into permanent silence.

## Acceptance Notes
- Snapshot still reads one prepared surface VM path
- app does not compute thresholds or visibility locally
- persistence stays narrow and scoped to reorientation dismissal only
- service-owned visibility logic remains explicit and explainable
