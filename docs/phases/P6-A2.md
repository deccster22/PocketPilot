# P6-A2 - Guarded Stop / Referral Surface Rollout

## Why This Phase Happened Now
P6-A1 established one canonical message-policy seam and made the message families explicit.
The next natural rollout step was not "more messages everywhere."
It was attaching two already-modeled families to the surfaces where they naturally belong:
- Dashboard `REFERRAL`
- Trade Hub `GUARDED_STOP`

This phase happened now to deepen the seam without turning it into a notification platform.

## What This Phase Added
- one unchanged canonical message-policy contract with only small additive interpreted-context refinement upstream of classification
- one canonical `fetchMessagePolicyVM` entry point that now serves:
  - Snapshot
  - Dashboard
  - Trade Hub
- one Dashboard inline referral note when Dashboard has supporting interpreted context but no prime focus item
- one Trade Hub inline guarded-stop note when the selected confirmation session has no protected execution path
- thin screen-facing helpers so `app/` renders prepared policy output only

## Distinctions Preserved
P6-A2 keeps the lines from P6-A1 explicit in actual rendered surfaces:
- `BRIEFING` remains quieter than `ALERT`
- `REORIENTATION` remains separate from `ALERT`
- `REFERRAL` remains a routing or fit note, not a hard failure
- `GUARDED_STOP` remains a product-boundary stop, not a readiness warning
- thin or missing context still resolves to `UNAVAILABLE`, not filler copy

## Deliberately Not Added
P6-A2 does not add:
- push notifications
- background polling
- inbox or unread systems
- badge counts
- notification center plumbing
- contextual knowledge rails
- generic toast or banner infrastructure
- broad all-surface rollout
- execution or order dispatch

## What Future P6 Work Can Build On
Later `P6` work can build on this phase by:
- adding more surface-specific eligibility only where a natural inline home exists
- refining alert thresholds with richer interpreted context
- deepening referral or guarded-stop inputs without moving classification into `app/`

Any future work should keep the same canonical message-policy seam and avoid drifting into a notification platform.

## Recommendation Review
### Adopt Now
- Dashboard `REFERRAL` rollout: it is the natural inline fit when Dashboard has supporting context but no prime focus item.
- Trade Hub `GUARDED_STOP` rollout: it cleanly reuses confirmation-capability truth to mark a product boundary without implying execution.

### Add To Backlog
- Broader per-surface rollout: useful later, but this phase deliberately stops at Snapshot, Dashboard, and Trade Hub.
- Richer alert-threshold tuning: valuable once more interpreted contexts need message-policy shaping, but not required for this thin rollout.

### Decline For Now
- Push, inbox, and badge mechanics: they conflict with the calm inline policy posture and would expand scope into a notification system.
- Generic toast or banner systems: they would blur message-family distinctions and encourage cross-surface theatre instead of product-specific placement.
