# P8-I10 - Export Dispatch + Journal-Aware Follow-through

## Why This Phase Happened Now

P8-I8 added one calm export-preparation seam over the existing reflection family.
P8-I9 added one calm journal-entry groundwork seam for period and annual reflection context.

That made the next safe step clear:

- keep deepening the same Insights reflection lane instead of starting a reporting product
- add one canonical export-dispatch seam on top of the prepared export contract
- allow narrow journal-aware follow-through only where the current reflection context and export format can support it honestly

P8-I10 happens now because PocketPilot already had enough prepared reflection structure to create a real user-requested export file without widening into background work, analytics theatre, or journal sprawl.

## What P8-I10 Added

P8-I10 adds six narrow refinements:

1. One canonical export-dispatch contract family in `services/insights/types.ts` for dispatch availability, prepared export documents, and dispatch results.
2. One deeper prepared export seam in `services/insights/createPreparedExportRequest.ts` and `services/insights/fetchPreparedExportRequest.ts` that now carries dispatch availability plus explicit journal follow-through only when it is honestly supported.
3. One canonical dispatch seam in `services/insights/dispatchExportRequest.ts`.
4. One deterministic file renderer in `services/insights/renderPreparedExportFile.ts` plus one small filesystem adapter in `providers/insightsExportDispatchProvider.ts`.
5. One minimal real create-export path inside the existing `InsightsExportScreen` flow.
6. One narrow linked-summary-note follow-through path for PDF summary exports only when the current period context already has a journal note and the user explicitly includes it.

The read and create path is now:

`EventLedger + linked period journal context -> createPreparedExportRequest / fetchPreparedExportRequest -> InsightsExportScreen -> dispatchExportRequest -> insightsExportDispatchProvider`

## Reflection And Export Rules Locked In Here

P8-I10 establishes these rules:

- one canonical export-dispatch contract only
- one canonical dispatch seam only
- prepared export shaping stays in `services/`
- file rendering stays deterministic once inputs are supplied
- `app/` reads prepared request and dispatch result contracts only, then triggers one explicit user action
- exports stay calm, descriptive, explicit, and profile-aware
- dispatch remains user-triggered only; there is no silent or background export work
- linked journal follow-through stays narrow and explicit
- linked journal text is never auto-included unless the prepared contract supports it and the user selects it
- journal-aware follow-through is limited to the current period-summary PDF path in this phase
- timezone labeling remains visible in the user-facing export path
- provider/runtime diagnostics, raw signal codes, and internal-only metadata stay out of user-facing export output

## What P8-I10 Deliberately Did Not Add

P8-I10 does not add:

- a broad export-management center
- background export jobs
- push notifications
- AI-generated commentary
- gamified recap or report-card flows
- scorecards, streaks, or shame language
- broad journal browsing or journal export history
- bulk note export
- app-side export payload shaping
- market or order dispatch of any kind

## What Future Reflection Work Can Build On

Later `P8` work can safely extend this seam with:

- broader but still explicit file-share support when platform handling is ready
- richer archive-to-export source selection if reflection shelves widen later
- additional journal-aware export follow-through for other prepared reflection contexts when those contexts gain explicit retrieval rules
- calmer access to previously created exports if PocketPilot later needs a light export history shelf

Those future steps should deepen the same prepared-contract and service-owned dispatch seam rather than turning Insights into Reporting HQ, a notes product, or an analytics theatre surface.

## Recommendation Review

- Adopt now: One canonical export-dispatch seam plus one narrow PDF-linked journal follow-through path because P8-I8 and P8-I9 already established the reflection contracts needed to support a real file creation path honestly.
- Add to backlog: Broader share support, calmer created-export retrieval, and wider context-linked journal follow-through because they may become useful later but are not required for this first explicit dispatch phase.
- Decline for now: Export centers, background export automation, AI reflection commentary, bulk journal export, and gamified recap flows because they would turn calm reflection into management theatre or pressure.
