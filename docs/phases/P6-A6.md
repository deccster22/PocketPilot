# P6-A6 - Alert / Message Policy Consolidation

## Why This Phase Happened Now

P6-A1 established one canonical message-policy seam.
P6-A2 proved that the same seam could serve Snapshot, Dashboard, and Trade Hub without becoming a notification platform.
P6-A3 made alert posture profile-aware.
P6-A4 improved the interpreted inputs that feed the same seam.
P6-A5 added the calm user-visible rationale path on top of that same policy result.

That left one useful cleanup rung: keep the same behavior, but compact the policy and rationale outputs into one grouped prepared lane so the current consumers can thread a single service-owned contract instead of scattered ad hoc fields.

This phase happened now because the lane was stable enough to consolidate without changing product posture, and because future P6 work is easier to keep merge-safe when the current message path is grouped explicitly.

## What Was Consolidated

- one canonical `MessagePolicyLane` contract in `services/messages/types.ts`
- one service-owned lane composer in `services/messages/createMessagePolicyLane.ts`
- one canonical fetch seam that now returns the grouped lane for Snapshot, Dashboard, and Trade Hub
- one app-facing read path on each current consumer surface that reads the grouped lane only
- one preserved calm rationale path that stays paired with the same policy result

## What Was Deliberately Not Added

- no push notifications
- no inbox or unread systems
- no badge mechanics
- no notification center
- no background polling
- no delivery-platform widening
- no user-editable message settings maze
- no app-owned classification, tuning, or rationale derivation
- no hidden automation

## What Future P6 Work Can Build On

Later `P6` work can build on this phase by:
- adding more calm inline message surfaces only where a natural home already exists
- refining copy for the current families without changing ownership boundaries
- reusing the same grouped lane instead of inventing separate policy and rationale paths

Future work should still keep the family explicit, service-owned, and foreground-only.

## Verification

- added coverage for grouped lane composition and determinism
- preserved current policy, tuning, and rationale semantics
- updated the current surface-view tests to expect lane consumption
- `npm run verify` remains the merge gate for this phase

## Recommendation Review

### Adopt Now

- Canonical `MessagePolicyLane` composer: adopt now because the family is already stable and now needs compaction.
- Grouped surface threading through Snapshot, Dashboard, and Trade Hub: adopt now because it reduces duplication without changing the product posture.
- Calm app consumption of the grouped lane: adopt now because the UI should keep rendering prepared contracts only.

### Add To Backlog

- Broader rollout to any future message-consuming surfaces that are not yet in the current family
- Small presentation refinements if they stay subordinate and calm
- Any later lane enrichment that does not change the current foreground-only discipline

### Decline For Now

- Push notifications
- Inbox, unread, badge, and notification-center mechanics
- Background polling or background delivery
- A broader settings maze
- Hidden automation or delivery-platform behavior
