# P6-A4 - Richer Interpreted Alert Inputs

## Why This Phase Happened Now
P6-A1 established one canonical message-policy seam.
P6-A2 proved the same seam could serve Snapshot, Dashboard, and Trade Hub without becoming a notification platform.
P6-A3 made alert treatment profile-aware and explicit about keep, downgrade, or suppress behavior.

The next natural step was not more surfaces, more delivery mechanics, or user settings.
It was improving the truth quality of the interpreted inputs already feeding the seam so alert-versus-briefing decisions could stay calm, deterministic, and service-owned.

This phase happened now because richer interpreted context was the clearest follow-on recommendation from P6-A3 and the lowest-risk way to deepen the family without widening scope.

## What This Phase Added
- one additive refinement in `services/messages/types.ts` for `PreparedMessageInputContext` plus compact context enums
- one canonical helper in `services/messages/createPreparedMessageInputs.ts`
- one tighter Snapshot alert path in `services/messages/createMessagePolicyVM.ts`
- one unchanged canonical `fetchMessagePolicyVM` seam that now passes enough service-owned interpreted context for richer derivation
- focused tests proving the richer input model stays deterministic, conservative, and free of raw leakage

## Richer Interpreted-Input Refinement Added
P6-A4 adds one small prepared context shape for Snapshot alert policy:

- `subjectScope`
  Why: alert treatment should know whether the interpreted change is about one symbol, a small symbol cluster, or portfolio-level context.
- `eventFamily`
  Why: the message seam should reason over compact families such as price change, momentum, or pullback rather than re-deriving meaning ad hoc.
- `confirmationSupport`
  Why: the seam should distinguish a confirmed standalone event from confirmed change that is also supported by interpreted continuity such as Since Last Checked or reorientation context.
- `changeStrength`
  Why: the seam should use one compact interpreted strength scale instead of scattering raw confidence and magnitude checks across message-building logic.
- `isSingleSymbolScope`, `hasSinceLastCheckedContext`, and `hasReorientationContext`
  Why: these compact service-owned flags keep policy readable and conservative without leaking raw event payloads.

The helper derives those fields from existing interpreted seams only:
- latest relevant interpreted market event
- Snapshot briefing state
- reorientation state
- Since Last Checked summary count

It does not consume raw signal arrays to create a second interpretation engine.
It clarifies existing service truth instead.

## How Policy Changed Without Changing The Family Model
P6-A4 does not add new message families.
It improves the quality of input the existing family model can evaluate.

Current policy effect:
- strong, single-symbol, confirmed change can still remain an `ALERT`
- meaningful but not fully strong change can stay quiet unless interpreted history support honestly confirms it
- multi-symbol or broader-scope change stays conservative and can downgrade to `BRIEFING` instead of becoming a louder alert
- thin or estimated context still suppresses to no message
- `REORIENTATION`, `REFERRAL`, and `GUARDED_STOP` keep their existing meanings and precedence

This phase is about better truth, not more noise.
Richer context can confirm suppression just as easily as it can preserve an alert.

## Deliberately Not Added
P6-A4 does not add:
- push notifications
- background polling or background delivery
- inbox or unread systems
- badge counts
- notification center plumbing
- user-editable sensitivity settings
- broader surface rollout beyond Snapshot, Dashboard, and Trade Hub
- raw signal exposure
- delivery-platform behavior
- AI-generated copy
- urgency escalation mechanics

## What Future P6 Work Can Build On
Later `P6` work can build on this phase by:
- refining other naturally available interpreted inputs on the same seam
- extending policy-aware eligibility only where a calm inline surface already exists
- improving calm explanatory copy for existing families

Future work should still reuse the same message-policy seam and avoid turning PocketPilot into a notification platform.

## Recommendation Review
### Adopt Now
- Prepared alert-input context helper: it improves policy truth with one canonical service-owned derivation path.
- Single-symbol versus multi-symbol scope handling: it keeps alert posture conservative when the interpreted subject is broader than one symbol.
- History-backed confirmation support: it lets the seam distinguish stronger interpreted continuity from thin standalone change without adding delivery mechanics.

### Add To Backlog
- Additional surface eligibility rules on the same seam: useful later, but this phase deliberately keeps rollout fixed to Snapshot, Dashboard, and Trade Hub.
- User-visible explanation of why an alert was downgraded or suppressed: potentially helpful later, but unnecessary for this thin truth-focused refinement.
- Further prepared context fields beyond the current compact set: valuable only if they remain high-trust and avoid recreating an interpretation engine.

### Decline For Now
- Push, inbox, and badge mechanics: they would widen P6-A4 into a notification platform instead of an input-refinement phase.
- User-editable sensitivity settings: they would move tuning into settings UX before the service-owned policy seam is mature enough.
- Generic toast or banner infrastructure: it would blur message-family semantics and encourage louder delivery behavior.
