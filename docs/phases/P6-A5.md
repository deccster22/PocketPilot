# P6-A5 - User-Visible Message Rationale

## Why This Phase Happened Now
P6-A1 established one canonical message-policy seam.
P6-A2 proved that the same seam could serve Snapshot, Dashboard, and Trade Hub without becoming a notification platform.
P6-A3 made alert posture profile-aware.
P6-A4 improved the interpreted inputs that feed the same policy seam.

That left one important trust gap: users could see a message posture, but PocketPilot still had no single calm way to explain why that posture surfaced as a briefing, alert, referral, guarded stop, or reorientation note.

This phase happened now because the next useful rung was not push, inbox, badge, or settings work.
It was one compact explanatory seam that deepens legibility while keeping the broader P6 family calm, service-owned, and inline.

## What This Phase Added
- one canonical `MessageRationale` contract plus explicit `MessageRationaleAvailability` in `services/messages/types.ts`
- one canonical builder in `services/messages/createPreparedMessageRationale.ts`
- one unchanged canonical `fetchMessagePolicyVM` entry path, now returning prepared rationale alongside the prepared message-policy result
- one additive refinement in `services/messages/createMessagePolicyVM.ts` so classification and rationale shaping come from the same service-owned pass
- one minimal inline rationale treatment on Snapshot, Dashboard, and Trade Hub through prepared screen-view contracts and `app/components/MessageRationaleNote.tsx`
- focused tests proving rationale remains deterministic, compact, calm, and free of raw signal/provider/runtime leakage

## Rationale Support Added
The new rationale seam is deliberately small.

It answers:
- why this message family was chosen
- why the message stayed a briefing instead of becoming an alert where relevant
- why the note stays compact and inline on the current surface

It does not answer:
- raw signal names
- scoring internals
- provider/runtime diagnostics
- what the user should do
- why a future push or badge might exist, because those systems are still out of scope

Current rationale posture examples include:
- alert because the change is focused enough for Snapshot's alert posture
- briefing because the context is broader or calmer than PocketPilot uses for alert treatment
- referral because Snapshot is the steadier first read
- guarded stop because Trade Hub should keep the current boundary visible

## Deliberately Not Added
P6-A5 does not add:
- push notifications
- inbox, unread, or badge mechanics
- notification-center views
- background polling
- generic toast or banner infrastructure
- user-editable sensitivity settings
- raw provider/runtime diagnostics for users
- advice, prediction, or urgency language
- app-owned message classification or rationale derivation

## What Future P6 Work Can Build On
Later `P6` work can build on this phase by:
- extending rationale coverage only where an existing calm inline message surface already exists
- refining copy for current message families without changing ownership boundaries
- reusing the same prepared rationale seam for future message-family deepening instead of inventing separate explanatory paths

Future work should still avoid turning PocketPilot into a delivery platform, debug console, or settings maze.

## Recommendation Review
### Adopt Now
- One canonical user-visible rationale seam: it improves trust without widening the product surface area.
- Service-owned rationale shaping: it keeps explanation tied to the same classifier and tuning pass that chose the message posture.
- Small inline rationale treatment on existing message surfaces: it gives users a calm "why this is here" path without introducing a new navigation or delivery system.

### Add To Backlog
- Broader rationale coverage for future message-consuming surfaces, but only if they already have a natural inline message lane.
- Additional calm copy refinements for current families once real product usage shows which explanations need the most tightening.
- Any future quiet explanation for fully suppressed posture, but only if it can be surfaced without inventing a notification center or debug view.

### Decline For Now
- Push, inbox, unread, badge, and notification-center mechanics: they would widen the phase into platform work instead of trust-legibility work.
- User-editable sensitivity controls: they would create a settings maze before the policy family is fully mature.
- Raw diagnostics or scoring disclosure: they would turn rationale into a debug panel instead of a user-facing explanation seam.
