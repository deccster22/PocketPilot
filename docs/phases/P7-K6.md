# P7-K6 - Knowledge Topic-Linkage Expansion

## Why this phase happened now

P7-K4 proved the canonical contextual-knowledge lane on Dashboard and Trade Hub. P7-K5 refined how much of that lane should appear and when it should quietly disappear. P7-K6 happens now because the lane is stable enough to improve the quality of the topic linkage itself without changing the K4/K5 presentation model or moving any selection logic into `app/`.

## What was improved

- one canonical contextual topic-linkage seam in `services/knowledge`
- one deterministic service-owned ranking path that can use strategy, signal, event, and surface context
- one better prepared topic selection result for Dashboard and Trade Hub
- one preserved K4/K5 presentation seam so the shelf still stays optional, calm, and subordinate

## What was deliberately not added

- no push notifications
- no background polling
- no forced tutorials
- no execution lockouts
- no gating based on knowledge completion
- no giant recommendation feed
- no app-owned topic selection logic
- no Snapshot expansion
- no inbox, badge, or campaign mechanics

## What future P7 work can build on top of it

- additional contextual surfaces only when they have honest relevance
- richer prepared topic-detail follow-through without moving ranking into `app/`
- future density tuning that stays subordinate to the current surface
- stronger contextual links only if they continue to preserve the calm, optional presentation contract

## Recommendation handling

- Adopt now: yes, the linkage seam is worth keeping because it improves topic quality without changing rollout posture
- Add to backlog: any future surface expansion or deeper detail follow-through that depends on the same linkage seam
- Decline for now: push, inbox, background monitoring, execution gating, or a recommendation feed
