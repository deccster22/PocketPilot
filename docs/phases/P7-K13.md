# P7-K13 - Knowledge Catalog Runtime Sync Guard

## Why This Phase Happened Now

`P7-K12` shipped the first bounded Trade Hub/Risk Tool help rollout and exposed a routing drift class:

- topic IDs existed in `docs/knowledge/_register/CONTENT_REGISTER.csv`
- the same IDs were missing from runtime `services/knowledge/knowledgeCatalog.ts`
- taps reached Knowledge Topic detail but returned `TOPIC_NOT_FOUND`

`P7-K13` adds deterministic tooling guardrails so this drift class fails validation before merge.

## Dependencies

- `PX-KI3` for register validation baseline.
- `PX-KI4` and `PX-KI5` for the active register/content families and Trade Hub glossary/topic imports.
- `P7-K12` for the first bounded runtime Trade Hub term-help routing scope.

## What Validation Now Prevents

- explicit runtime-required register topic IDs drifting out of runtime `knowledgeCatalog`
- duplicate runtime `topicId` entries in `knowledgeCatalog`
- hidden runtime-only topic IDs that are not in register and not explicitly allowlisted as legacy exceptions
- stale legacy allowlist entries that are no longer present at runtime

## Explicit Runtime Scope Model

Runtime-sync validation now uses an explicit scope model:

- `RUNTIME_REQUIRED_REGISTER_TOPIC_IDS` lists registered topics that must resolve at runtime.
- `RUNTIME_ALLOWED_UNREGISTERED_TOPIC_IDS` lists legacy runtime IDs that remain unregistered by design for now.
- all other register rows are treated as docs-only/non-runtime unless promoted into the required runtime list.

This keeps exclusions explicit rather than silently skipping rows.

## What Was Deliberately Not Changed

- no app UI copy, layout, or tap placement changes
- no Trade Hub/Risk Tool runtime behavior changes
- no risk math, guardrail semantics, readiness, submission, or handoff semantic changes
- no new knowledge content imports
- no broad runtime catalog regeneration/rewrite
- no CANON edits
