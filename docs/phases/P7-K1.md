# P7-K1 - Knowledge Baseline Foundation

## Why This Phase Happened Now
PocketPilot already had the surface and interpretation groundwork needed for a first knowledge baseline:
- event and orientation seams in `P3`
- Snapshot and Dashboard shaping in `P4`
- Trade Hub action framing in `P5`
- early explanation groundwork in `PX-E1` and `PX-E2`

P7-K1 happens now because the product finally has enough canonical interpreted language to support a small knowledge layer without falling back to generic education content, raw signal leakage, or UI-owned assembly.

This phase establishes the substrate first so later contextual linking has something stable to point at.

## What P7-K1 Added
P7-K1 adds five baseline pieces:

1. One canonical `KnowledgeNode` contract in `services/knowledge/types.ts`.
2. One small PocketPilot-native seed catalog in `services/knowledge/knowledgeCatalog.ts`.
3. One service-owned library shaping seam in `services/knowledge/createKnowledgeLibraryVM.ts`.
4. One app-facing library fetch seam in `services/knowledge/fetchKnowledgeLibraryVM.ts`.
5. One minimal top-level Knowledge Library surface in `app/screens/KnowledgeLibraryScreen.tsx`.

The current path is:

`knowledgeCatalog -> createKnowledgeLibraryVM -> fetchKnowledgeLibraryVM -> KnowledgeLibraryScreen`

## Optional And Non-Gating Rules
P7-K1 locks in the product posture for knowledge:
- the library is always reachable as a top-level destination
- knowledge remains optional
- unavailable knowledge degrades honestly instead of blocking flows
- Dashboard, Snapshot, and Trade Hub do not depend on knowledge consumption
- app renders prepared library sections and items only

## What P7-K1 Deliberately Did Not Add
P7-K1 does not add:
- a full content-management system
- a giant educational corpus
- AI tutoring or chat-style explanation
- search, ranking, or recommendation complexity
- contextual-link rollout across multiple surfaces
- quizzes, streaks, or progress theatre
- background polling or push notifications
- trading execution

## Why The Phase Is Narrow
The baseline knowledge object and library seam need to become stable before PocketPilot multiplies knowledge touchpoints.

That is why P7-K1 prefers:
- a small strong catalog over placeholder volume
- one surface over many
- service-owned grouping over UI assembly
- optional availability over product gating

## What Future Work Can Build On
Future work can extend this baseline in a controlled way.

- Later `P7` work can add more research-backed PocketPilot knowledge nodes, richer detail presentation, and broader media coverage.
- `P8` work can reuse the same knowledge substrate when reflection, history, and reorientation need supportive reference material.
- `P9` work can connect richer explanation, pattern navigation, and strategy understanding back to the same canonical knowledge objects.

Those later phases should build on the current contracts instead of creating competing knowledge paths.
