# P5-R2 - Quote-Assisted Risk References

Status: done  
Last updated: 2026-04-05

## Why This Phase Happened Now

P5-R1 established the first canonical risk-tool seam, but it intentionally stopped short of quote-assisted entry support.

That pause was correct at the time.
PocketPilot needed a clean, non-leaky prepared quote context before the risk tool could consume current pricing without turning into order-entry sprawl or provider-explainer clutter.

P5-R2 happened now because the P5 and PX-API seams are mature enough to let services prepare a quote reference safely while keeping the tool:

- support-only
- deterministic once inputs are supplied
- explicit about what came from the user versus prepared context
- non-dispatching

## What This Phase Added

- one additive refinement of `services/risk/types.ts` so entry, stop, and target references are source-tagged
- one canonical `selectRiskReferences` seam in `services/risk/` that chooses explicit user values first, then honest prepared references
- one quote-assisted extension of `fetchRiskToolVM` that sanitizes prepared quote context into a current-price entry reference when available
- one update to `createRiskToolVM` so calculations run from source-tagged references without changing the P5-R1 formulas
- one calm Trade Hub rendering update that shows whether a reference came from user input, a current reference, a prepared plan, or nowhere yet

## What Quote-Assisted Risk Support Was Added

P5-R2 adds prepared reference support, not a new calculator family.

The risk tool can now use:

- explicit user entry, stop, and target values
- a prepared current-price quote as a candidate entry reference
- prepared stop or target references only when a service-owned plan context honestly provides them

The reference output stays explicit:

- `USER_INPUT`
- `PREPARED_QUOTE`
- `PREPARED_PLAN`
- `UNAVAILABLE`

That lets the app render calm source labels without exposing provider IDs, timestamps, routing paths, or signal metadata.

## What Was Deliberately Not Added

- no broker API integration
- no order dispatch
- no execution payload construction in `app/`
- no one-tap execution path
- no broker-style advanced calculator
- no leverage, margin, derivatives, or simulation theatre
- no hidden defaults or automation
- no invented stop or target values when prepared context is thin
- no raw provider, signal, or runtime leakage into the user-facing risk output

## What Future P5 Risk-Tool Work Can Build On

- richer prepared plan references once the Trade Hub family has a canonical service-owned way to carry them
- account-scoped risk preferences once a settings seam exists
- additional calm planning support around targets or multi-leg framing if a later phase explicitly expands scope

## Verification

- added canonical selector coverage for explicit override, prepared quote labeling, unavailable stop/target honesty, and opt-out behavior
- updated risk-tool VM coverage for source-tagged references, prepared quote plus plan mixes, incomplete states, and deterministic replay
- updated fetch coverage for prepared quote orchestration and no-leak behavior
- updated app view and placement coverage so the screen consumes prepared source labels without owning calculations or source choice
- `npm run verify` remains the merge gate for this phase

## Recommendation Review

### Adopt now

- Source-tagged references in the canonical risk contract: adopted now because trust depends on clearly separating user input from prepared assistance.
- One selector seam for all reference choice: adopted now because it keeps source precedence out of `app/` and avoids inconsistent future paths.
- Prepared quote assistance for entry only: adopted now because it adds useful context without inventing exits or implying execution readiness.

### Add to backlog

- Prepared stop and target references sourced from richer trade context: backlog, because the selector now supports them but the current Trade Hub plan/session seam does not yet carry honest values.
- Account-scoped preference toggles for prepared references: backlog, because explicit override exists today and a broader preference surface needs its own settings phase.
- Broader planning tools such as multi-target framing: backlog, because they may be valuable later but would widen this thin trust-preserving phase.

### Decline for now

- Broker-style order calculators: decline for now because they blur the support seam into order-entry behavior.
- Quote-derived stop or target suggestions: decline for now because the phase explicitly avoids hidden heuristics and invented exits.
- Execution readiness coupling inside the risk tool: decline for now because `RiskToolVM` and the execution stack must remain separate authoritative seams.
