# P7-K11 - Trade Hub Term-to-Knowledge Integration Plan

## Why This Phase Happened Now

`P5-R15` and `P5-R16` stabilized plain-language Trade Hub and Risk Tool terminology around planning levels, risk framing, and non-dispatch boundaries.

`PX-KI4` then imported the Trade Hub knowledge family, glossary set, and term-map support docs, and `PX-KI5` merged beginner-facing strategy/concept learning layers into canonical knowledge docs with deterministic validation coverage.

`P7-K8`, `P7-K9`, and `P7-K10` already established service-owned inline glossary behavior, alias normalization, and internal tuning hooks. The next honest step was a bounded integration plan that decides what should be help-eligible before any broader runtime wiring.

## Content Foundations This Phase Depends On

- `P5-R15` and `P5-R16` for current Trade Hub and Risk Tool user-facing labels.
- `PX-KI4` for Trade Hub/glossary knowledge import and term-map baseline.
- `PX-KI5` for progressive beginner/intermediate/advanced knowledge posture.
- `P7-K8`, `P7-K9`, and `P7-K10` for service-owned inline glossary architecture and matching/tuning continuity.

## What Integration Plan Was Created

- Added `docs/knowledge/_meta/TRADE_HUB_HELP_INTEGRATION_PLAN.md` as the canonical planning artifact for Trade Hub term help integration.
- Classified each required Trade Hub/Risk Tool term with:
  - glossary topic ID/path
  - full Knowledge topic ID/path
  - interaction mode (`none`, `glossary only`, `glossary first, then Knowledge`, `direct Knowledge topic`)
  - eligible profiles and surfaces
  - priority, rationale, and over-link/confusion risk
- Defined a deliberately small first-rollout candidate set for the next wiring phase:
  - `Stop-loss price`
  - `Target price`
  - active risk-basis label (`Risk amount` or `Risk percent`)
  - `Guardrails`
- Defined explicit no-link-soup rules and a service-owned future interaction pattern so `app/` remains render-only.
- Reconciled mapping continuity against existing term-map and UI-support docs and recorded known ambiguities for the next phase.

## What Was Deliberately Not Wired

- No app runtime wiring, no new tap affordances, and no UI-side term matching.
- No Trade Hub/Risk Tool visible copy, layout, or hierarchy changes.
- No risk calculations, prepared planning semantics, guardrail semantics, or submission/readiness behavior changes.
- No content import expansion, no second knowledge tree, and no Trade Hub article duplication.
- No runtime use of docs-side UI-support drafts.

## Recommended P7-K12 Scope

- Implement one service-owned static eligibility/routing map for the first-rollout term set only.
- Keep mapping and routing logic in `services/knowledge`; keep `app/` on prepared affordance consumption only.
- Respect non-linking rules from the P7-K11 plan:
  - avoid repeated linking
  - avoid dense numeric-summary linking
  - keep handoff boundary language plain in critical readiness/submission areas
- Route lightweight help through existing K8/K9 glossary paths and route deeper topics through existing Knowledge Topic detail paths.
- Defer broader term coverage and richer affordances until first-rollout clarity and clutter impact are reviewed.

## Recommendation Handling

### Adopt now

- Use the P7-K11 plan as the canonical decision baseline for term eligibility, priority, and no-linking guardrails.
- Keep later runtime work bounded to service-owned mapping plus prepared affordance output.

### Add to backlog

- Wider term coverage beyond the first-rollout candidate set.
- Broader surface expansion (for example Snapshot or multi-surface repeated linking patterns) after evidence-based review.

### Decline for now

- UI-side term matching/ranking.
- Link-everything or tooltip-swarm behavior.
- Any help pattern that weakens non-dispatch boundary language in submission/readiness lanes.
