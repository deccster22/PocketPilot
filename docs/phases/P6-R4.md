# P6-R4 - Snapshot Foreground Refresh Check

## Intent
P6-R4 keeps Snapshot truthful after a user backgrounds the app and comes back later in the same session.

This phase adds one calm foreground-only refresh trigger and keeps all Snapshot truth on the existing prepared `SnapshotSurfaceVM` path.

## Delivered
- Snapshot refreshes on initial mount through the existing prepared surface path.
- Snapshot refreshes again only when app lifecycle transitions from `background` or `inactive` to `active`.
- The refresh still flows through `services/snapshot/fetchSnapshotSurfaceVM.ts`.
- Reorientation visibility, dismissal honoring, and stale-dismiss clearing remain service-owned.
- Current-session dismissal now follows the same cycle boundary as persisted dismissal, so a newer eligible cycle may reappear without a cold restart.

## Non-Goals Preserved
- no order dispatch
- no broker integration
- no push notifications
- no background polling
- no notification center or inbox behavior
- no raw signal leakage
- no second Snapshot fetch seam

## Implementation Notes
- `app/screens/SnapshotScreen.tsx` listens for explicit app lifecycle return and does not use navigation focus churn as a refresh trigger.
- `app/screens/snapshotScreenView.ts` owns the thin screen refresh helper so mount and foreground return share one prepared VM path.
- `services/orientation/reorientationPersistence.ts` now treats current-session dismissal as cycle-aware, not as an unbounded boolean.

## Verification Focus
- initial mount fetch still works
- background or inactive to active triggers one refresh
- active-to-active churn does not trigger refresh
- reorientation remains hidden for the same dismissed cycle
- a newer eligible reorientation cycle reappears through the same shared service path
