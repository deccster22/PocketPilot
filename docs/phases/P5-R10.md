# P5-R10 - Opt-In Guardrail Preferences Foundation

## Why This Phase Happened Now

P5-R6 made the risk basis explicit and carried a prepared risk-per-trade context through Trade Hub. P5-R7 added prepared sizing and max-loss output. P5-R8 remembered an account-level preferred basis. P5-R9 added calm guidance for thin or unsupported inputs.

P5-R10 happened now because the Trade Hub lane was finally stable enough to remember one narrow set of optional guardrail preferences without turning PocketPilot into a default-blocking enforcement system or a broad settings maze.

## What Was Added

- one canonical `GuardrailPreferences` contract family in `services/trade/types.ts`
- one canonical account-scoped guardrail-preferences store in `services/trade/guardrailPreferencesStore.ts`
- one canonical fetch seam in `services/trade/fetchGuardrailPreferences.ts`
- one canonical update seam in `services/trade/updateGuardrailPreferences.ts`
- one Trade Hub surface contract field that carries guardrail-preferences availability alongside the other prepared risk-lane truth
- one calm Trade Hub proof path that shows optional guardrail state and explicit update actions without adding enforcement or blocking by default

## What Was Deliberately Not Added

- no live order dispatch
- no hard enforcement by default
- no auto-blocking guardrails
- no background polling
- no push notifications
- no portfolio-wide analytics or scorecards
- no broad settings center
- no app-owned persistence logic
- no fear-language, shame-language, or compliance theatre

## What Future Guardrail Work Can Build On

- richer preference editing surfaces if the product ever needs them explicitly
- durable persistence behind the current service-owned store seam
- future enforcement only through a separate explicit phase
- additional optional risk guardrails if they remain calm, explicit, and account-scoped

## Verification

- added direct coverage for explicit default-off fetch behavior, account-scoped isolation, and unavailable surface states
- added direct coverage for explicit updates, unchanged writes, invalid configuration rejection, and first-time explicit off updates
- updated Trade Hub surface and screen-view tests so the new guardrail-preferences contract stays prepared in services and rendered in the app without app-owned persistence
- `npm run verify` remains the merge gate for this phase

## Recommendation Review

### Adopt now

- one canonical guardrail-preferences contract: adopt now because optional guardrails need their own explicit service-owned shape
- one service-owned fetch/update seam: adopt now because the app should never own preference persistence rules
- one calm Trade Hub proof path: adopt now because the lane should show optional structure the user chose without introducing blocking behavior

### Add to backlog

- durable persistence behind the current in-memory store seam: add to backlog because the phase only needs a deterministic foundation today
- richer preference editing surfaces: add to backlog because the current proof path is intentionally narrow
- cross-surface reuse of guardrail preferences: add to backlog because this phase intentionally proves the path in Trade Hub first

### Decline for now

- enforcement-by-default: decline for now because PocketPilot stays support-first
- a broad settings maze: decline for now because it would fight the product's calm posture
- automatic blocking when guardrails are enabled: decline for now because the phase is explicitly non-enforcing
