# P5-R8 - Account-Level Preferred Risk Basis

Status: done
Last updated: 2026-04-12

## Why This Phase Happened Now

P5-R6 made the risk basis explicit and prepared the risk-per-trade context.
P5-R7 extended the prepared sizing and max-loss lane.

P5-R8 happened now because the next honest rung was to remember a user's usual basis per account so Trade Hub could start from a stable account-level preference without turning the product into a settings maze or an enforcement engine.

## What Was Added

- one canonical `PreferredRiskBasis` contract family in `services/trade/types.ts`
- one service-owned preferred-risk-basis store in `services/trade/preferredRiskBasisStore.ts`
- one canonical fetch seam in `services/trade/fetchPreferredRiskBasis.ts`
- one canonical update seam in `services/trade/updatePreferredRiskBasis.ts`
- Trade Hub fetch paths that seed the prepared risk lane from the saved preferred basis when it is honestly available
- one explicit Trade Hub surface note that shows the account's usual basis without hiding the active selection
- one minimal app proof path that updates the preferred basis only through the service seam

## What Was Deliberately Not Added

- no live execution
- no hard enforcement by default
- no portfolio-wide risk analytics
- no global preferred-basis default
- no cross-account sync rule
- no background polling
- no push notifications
- no broad settings/preferences center

## What Future Risk-Tool Work Can Build On

- a real persistence adapter behind the current service-owned store seam
- richer account-scoped preference surfaces if the product ever needs them explicitly
- deeper risk assistance that still begins from one honest per-account default
- future enforcement work only through a separate explicit phase

## Recommendation Review

### Adopt now

- one account-scoped preferred-basis seam: adopt now because the risk lane needed a stable starting point that stays explicit and service-owned
- one Trade Hub preference note: adopt now because the selected basis should stay legible and not feel hidden
- one service-owned update path: adopt now because the app should not own persistence rules

### Add to backlog

- a durable persistence adapter for preferred basis state: add to backlog because the current in-memory store is enough for the phase proof path
- richer preference management surfaces: add to backlog because the current goal was only one calm account-level seam

### Decline for now

- enforcement-by-default: decline for now because PocketPilot remains support-first
- global basis defaults: decline for now because they would blur account-scoped truth
- a broad settings maze: decline for now because it would fight the product's calm, narrow posture
