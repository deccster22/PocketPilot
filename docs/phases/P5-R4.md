# P5-R4 - Prepared Plan Producers for Risk References

Status: done  
Last updated: 2026-04-05

## Why This Phase Happened Now

P5-R1, P5-R2, and P5-R3 already established the calm risk-tool seam, quote-assisted entry help, and one selected-session handoff for prepared plan references.

P5-R4 happened now because the next honest improvement was not a wider contract or a new calculator family. It was better producer truth inside `services/trade/` so the existing risk seam could receive prepared entry, stop, and target references when the selected plan context already states them clearly.

## What Producer Enrichment Was Added

- added `services/trade/createPreparedRiskReferences.ts` as the one canonical producer helper for prepared plan risk references
- `createProtectionPlans` now uses that helper while shaping each `ProtectionPlan`
- the helper only emits values from service-owned trade context that already expresses them honestly:
  - one explicit positive `entryPrice`, `stopPrice`, or `targetPrice` from grouped `MarketEvent.metadata.preparedRiskReferences` when the grouped event context agrees on that field
  - one fallback prepared `entryPrice` from a confirmed primary-event price when the plan intent is actionable and no explicit prepared entry reference exists
- ambiguous or conflicting prepared stop/target/entry values stay `null`
- `fetchConfirmationSessionVM` now normalizes the prepared plan references from the selected plan through the existing selected-session handoff
- `selectRiskReferences` and `fetchRiskToolVM` keep the same boring precedence:
  - `USER_INPUT`
  - `PREPARED_PLAN`
  - `PREPARED_QUOTE`
  - `UNAVAILABLE`

## What Was Deliberately Not Added

- no broker APIs
- no order dispatch
- no execution payload construction in `app/`
- no quote-derived stop or target suggestions
- no hidden automation or defaults
- no account-scoped preferences or settings
- no multi-target reward framing
- no leverage, margin, derivatives, or P/L simulation tooling
- no one-tap or prefilled execution path

## What Future P5 Risk-Tool Work Can Build On

- richer strategy-owned prepared risk references when later service-owned plan context can express them explicitly
- account-scoped prepared-reference preferences once a dedicated settings seam exists
- broader planning support such as multi-target framing only if a later phase widens scope on purpose

## Verification

- updated `createProtectionPlans` coverage for actionable entry fallback, explicit prepared stop/target production, ambiguous stop/target null behavior, determinism, and no-leak behavior
- updated `createTradePlanPreview` coverage so preview shaping stays free of prepared risk-reference fields
- updated `fetchConfirmationSessionVM` coverage for selected-session handoff of producer-owned prepared risk references plus no-leak behavior
- updated `fetchRiskToolVM` coverage so prepared plan entry still outranks quote help while explicit user values remain authoritative
- `npm run verify` remains the merge gate for this phase

## Recommendation Review

### Adopt now

- Canonical trade-service prepared-risk helper: adopted now because one producer rule is easier to trust than ad hoc per-surface shaping.
- Confirmed primary-event price as fallback prepared entry for actionable plans: adopted now because it adds honest entry context without inventing exits.
- Preview contract staying free of prepared risk-reference fields: adopted now because source trust still belongs in services, not `app/`.

### Add to backlog

- Strategy-owned publishing of richer prepared stop/target references: add to backlog because the helper now supports honest inputs, but current strategy/event context still expresses only a thin set of values.
- Account-scoped prepared-reference preferences: add to backlog because explicit user override already exists and settings are out of scope here.
- Multi-target or laddered reward framing: add to backlog because it may help later but would widen this thin support seam.

### Decline for now

- Quote-derived stop or target suggestions: decline for now because the phase must not invent exits from thin quote context.
- Direct execution prefills from prepared plan references: decline for now because the risk lane must remain support-only and non-dispatching.
- Broker-style advanced calculators: decline for now because they would blur prepared planning support into order-entry sprawl.
