# P5-R7 - Position Sizing + Max-Loss Outputs

Status: done  
Last updated: 2026-04-12

## Why This Phase Happened Now

P5-R6 made the selected risk basis explicit and carried a prepared risk-per-trade context through the Trade Hub lane. P5-R7 happened now because the Trade Hub / ProtectionPlan stack was stable enough to add one dedicated prepared sizing/max-loss seam without widening into execution, enforcement, or portfolio-analytics sprawl.

## What Sizing And Max-Loss Support Was Added

- added one canonical `PositionSizingOutput` contract plus `PositionSizingAvailability` in `services/trade/types.ts`
- added one canonical `createPositionSizingOutput` builder in `services/trade/createPositionSizingOutput.ts`
- added one shared sizing snapshot helper in `services/trade/resolvePreparedSizingSnapshot.ts` so the math stays service-owned and deterministic
- threaded selected risk basis, prepared plan references, and selected-account context through:
  - `fetchTradePlanPreviewVM`
  - `fetchConfirmationSessionVM`
  - `TradePlanPreview`
- added one calm Trade Hub proof path:
  - a subordinate prepared sizing/max-loss block in the Trade Hub plan-preview lane
- kept the sizing lane support-first and non-enforcing
- kept the output explicit, calm, and read-only

## Support-First Rules Preserved

- no order dispatch
- no hidden execution logic
- no enforcement-by-default
- no app-side sizing math
- no profit-seeking or excitement language
- no auto-blocking guardrails
- no portfolio-wide VaR or advanced analytics

The new readout answers: what size and max loss does this framing imply?  
It does not answer: should the user take this trade?

## What Was Deliberately Not Added

- no live execution coupling
- no broker-specific payload logic
- no hard enforcement modes
- no broader portfolio-risk analytics
- no gamified risk/reward visuals
- no notification, polling, or background work
- no cross-surface rollout outside Trade Hub

## What Future Work Can Build On

- richer account-aware sizing defaults if a future settings seam makes user intent explicit
- additional prepared risk readouts once more plan producers publish honest references
- optional follow-through into knowledge or coaching surfaces without changing the Trade Hub boundary
- future enforcement or portfolio analytics only through a separate explicit phase

## Verification

- added direct coverage for available and unavailable sizing outputs
- updated preview and confirmation-session tests so sizing flows through prepared Trade Hub outputs
- updated Trade Hub preview view tests so the app only formats prepared sizing contracts and does not perform sizing math
- `npm run verify` passes

## Recommendation Review

### Adopt now

- one canonical prepared sizing/max-loss builder: adopted now because the support-only readout belongs in services, not the screen
- one shared sizing snapshot helper: adopted now because the max-loss math must stay deterministic and reusable
- one calm Trade Hub sizing proof path: adopted now because the user should see the implication of the prepared plan without a pressure surface

### Add to backlog

- richer account-aware sizing defaults: add to backlog because explicit user preferences deserve their own settings seam
- cross-surface reuse of this same sizing lane outside Trade Hub: add to backlog because this phase intentionally proves the path in one lane only

### Decline for now

- hard risk enforcement or auto-blocking: decline for now because PocketPilot remains support-first, not paternalistic
- broad portfolio-risk analytics: decline for now because that would widen the phase far beyond the Trade Hub seam
- profit-framed or gamified sizing storytelling: decline for now because it conflicts with PocketPilot's calm posture
