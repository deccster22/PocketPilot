# P5-R1 - Risk Tool / Position Sizing Foundation

Status: done  
Last updated: 2026-04-05

## Why This Phase Happened Now

P5 already had the core Trade Hub action stack:

- `ProtectionPlan`
- `TradeHubSurfaceModel`
- `ConfirmationSession`
- execution preview, readiness, submission-intent, and simulated adapter seams

What it did not yet have was a thin canonical risk-tool lane for disciplined position framing.

P5-R1 happened now to add that missing first rung without turning Trade Hub into an order ticket, broker calculator, or hype surface.

## What This Phase Added

- one canonical `services/risk/types.ts` contract for explicit risk inputs plus calm prepared summaries
- one deterministic `createRiskToolVM` builder that owns stop distance, risk amount, position size, and reward/risk framing
- one canonical `fetchRiskToolVM` seam that combines selected-plan context with explicit user inputs
- one Trade Hub-adjacent risk-tool section that renders prepared values only
- one top-level Trade Hub tab path so the action stack, including the new risk tool, is reachable in the current app shell
- deterministic tests that keep calculation logic in services and out of `app/`

## What Was Deliberately Not Added

- no order ticket
- no broker payload construction
- no submission or dispatch behavior
- no leverage, margin, or derivatives logic
- no portfolio analytics suite
- no P/L simulation theatre
- no hidden defaults, auto-fill, or automation
- no raw provider, signal, capability, or runtime metadata in user-facing risk output

## Foundation Rules Locked By This Phase

- services own all risk calculations
- app renders prepared contracts only
- incomplete inputs stay incomplete instead of inventing precision
- reward/risk may remain null when it cannot be framed honestly
- risk readiness is not execution readiness
- the tone stays factual and support-first

## What Future P5 Risk-Tool Work Can Build On

- account-scoped default risk preferences once a canonical settings seam exists
- quote-aware input assistance once entry references can be supplied without leaking provider/runtime detail
- additional calm planning tools around targets or multi-leg framing if product scope explicitly expands

## Verification

- added deterministic service tests for ready, incomplete, unavailable, and no-leak paths
- added app view tests proving the screen consumes prepared values without recalculating them
- added placement coverage so risk calculations remain outside `app/`
- `npm run verify` should remain the merge gate for this phase

## Recommendation Review

### Adopt now

- Keep risk tooling as a sibling seam to `ConfirmationSession`: this preserves the existing P5 architecture and avoids turning confirmation into a hidden calculator host.
- Require explicit user inputs for entry, stop, and risk basis: this keeps the summary honest and avoids fake certainty from inferred numbers.

### Add to backlog

- Account-scoped default risk preferences: valuable, but it needs a canonical settings lane rather than ad hoc local defaults.
- Quote-assisted entry and stop references: useful, but it should wait until there is a clean, non-leaky prepared quote reference seam for Trade Hub.
- Multi-target or laddered reward framing: plausible follow-on value, but it would expand the contract beyond the thin foundation intended here.

### Decline for now

- Broker-style advanced order calculators: too close to order-entry sprawl for this phase.
- Leverage or margin sizing: adds complexity that P5-R1 explicitly keeps out of scope.
- P/L simulation surfaces: invites theatre and distracts from calm disciplined framing.
