# P6-R2 - Reorientation Surface Placement

## Scope
P6-R2 attaches the existing reorientation summary seam to one canonical foreground surface through a shared upstream VM.

This phase does not expand reorientation into a notification product.

## Delivered In This Phase
- Snapshot is the one canonical foreground placement surface
- `services/snapshot/fetchSnapshotSurfaceVM.ts` prepares Snapshot core state together with reorientation placement state
- `services/orientation/createReorientationSurfaceState.ts` shapes explicit `VISIBLE` and `HIDDEN` outcomes
- lightweight dismissal is supported through explicit visibility input
- Snapshot renders only prepared placement state from services
- tests lock single-surface placement, hidden/visible behavior, dismissal, and deterministic shaping

## Behaviour Rules
- foreground-only
- one placement surface only
- subordinate inline card, not a modal or center-stage surface
- no broker APIs
- no order dispatch
- no push notifications
- no background polling
- no inbox/feed/badge system
- no guilt language
- no raw signal leakage

## Canonical Placement
Snapshot is the chosen home because it is already PocketPilot's calm foreground scan/orientation surface.

The card sits beneath the Snapshot core as a subordinate briefing element.
It must not crowd out the core triad.

## Visibility Contract
The shared surface VM carries explicit placement state:
- `VISIBLE` with `AVAILABLE`
- `HIDDEN` with `NOT_NEEDED`
- `HIDDEN` with `DISMISSED`

Dismissal is intentionally lightweight in this phase.
It hides the card without deleting the underlying summary contract and does not require durable persistence.

## Acceptance Notes
- app reads prepared Snapshot surface state only
- placement logic stays in `services/`
- reorientation remains optional and calm
- Snapshot remains the primary surface
- no inbox or notification system mechanics are introduced
