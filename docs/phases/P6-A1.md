# P6-A1 - Alert / Message Policy Foundation

## Why This Phase Happened Now
PocketPilot already had meaningful reorientation and briefing groundwork, but the broader `P6` family still lacked a canonical message-policy seam.

P6-A1 closes that gap without trying to finish the full alert family in one pass.

This phase happened now to:
- make message kinds explicit before broader rollout
- keep classification in `services/`
- protect PocketPilot's calm posture while `P6` is still early

## What This Phase Added
- one canonical `services/messages/types.ts` contract for product message policy
- one pure `createMessagePolicyVM` classifier
- one canonical `fetchMessagePolicyVM` service entry point
- one thin Snapshot consumer path that renders prepared message-policy output only
- explicit distinction between `BRIEFING`, `ALERT`, `REORIENTATION`, `REFERRAL`, and `GUARDED_STOP`

## Policy Foundations Added
The new foundation locks these lines in place:
- reorientation remains distinct from alerting
- quiet briefing remains distinct from alert-worthy change
- referral remains distinct from guarded-stop behavior
- `UNAVAILABLE` stays explicit when context is thin or a message belongs elsewhere
- app code does not classify messages locally

## Deliberately Not Added
P6-A1 does not add:
- push notifications
- background polling
- inbox or unread systems
- badges or notification-center plumbing
- a broad all-surfaces rollout
- AI-generated messaging
- raw signal exposure in user-facing message copy

## What Future P6 Work Can Build On
Later `P6` work can build on this phase by:
- adding explicit Trade Hub guarded-stop rendering
- adding explicit Dashboard referral rendering
- refining alert thresholds with more interpreted context
- expanding surface eligibility without moving classification into `app/`

Those future phases should reuse the same canonical message-policy model rather than branching into multiple message systems.
