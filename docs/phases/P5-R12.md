# P5-R12 - Trade Hub Risk Lane Consolidation

## Why This Phase Happened Now

P5-R6 through P5-R11 had already established the full support-first risk family: explicit basis, prepared risk-per-trade context, sizing and max-loss output, preferred basis memory, incomplete-input guidance, opt-in guardrail preferences, and calm guardrail evaluation.

P5-R12 happened now because that family was stable enough to compact into one cleaner prepared lane before future work adds any more surface area. The goal was consolidation, not new behavior.

## What Was Consolidated

- one canonical `TradeHubRiskLane` contract in `services/trade/types.ts`
- one service-owned lane composer in `services/trade/createTradeHubRiskLane.ts`
- one Trade Hub model path that carries the grouped prepared lane instead of scattering related risk fields across the surface model
- one app-facing view path that reads the grouped prepared lane without redoing any risk logic

## What Was Deliberately Not Added

- no new risk features
- no live order dispatch
- no hidden execution logic
- no hard enforcement by default
- no background polling
- no new settings center
- no app-owned risk math, persistence, or evaluation logic
- no fear-language, shame-language, or warning theatrics

## What Future P5 Work Can Build On

- a single canonical prepared risk lane for additional calm Trade Hub refinements
- service-owned additions that remain explicit, account-scoped, and non-blocking
- future guardrail or risk extensions that can reuse the grouped lane without reintroducing field sprawl
- any later enforcement work, if it ever exists, through a separate explicit phase rather than here

## Verification

- added coverage for grouped lane composition, preserved guardrail behavior, and calm app consumption
- updated Trade Hub surface and screen-view tests to expect the grouped prepared lane
- `npm run verify` remains the merge gate for this phase

## Recommendation Review

### Adopt now

- one clearer canonical Trade Hub risk-lane composition seam: adopt now because the family is already established and now needs compaction
- grouped surface threading through Trade Hub: adopt now because it reduces duplication without changing the product posture
- calm app consumption of the grouped lane: adopt now because the UI should keep rendering prepared contracts only

### Add to backlog

- any richer lane drill-downs that are not required for this consolidation
- cross-surface reuse of the same grouped lane beyond Trade Hub
- future optional presentation refinements if they stay calm and subordinate

### Decline for now

- new risk tools
- enforcement-by-default
- a broader settings maze
- any execution-path widening
